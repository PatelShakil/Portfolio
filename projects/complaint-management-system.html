Architecture at scale

Microservices communicate through Kafka events, fronted by an API gateway, orchestrated on Kubernetes for auto‑scaling and high availability.

Event‑driven Scalable Resilient

Distributed microservices with async messaging

Authentication

Unified JWT validation, RBAC enforcement, and rate‑limiting for citizens, field crews, and admins.

Complaint Service

Register, validate, persist to MySQL, and publish complaint.registered event to Kafka topic.

Assignment Service

Consume events, compute best‑fit crew via logic engine, store assignment, publish assignment.created.

Notification Service

Multi‑channel push via FCM, SMS gateway, email SMTP with delivery logs and retry logic.

Updates Service

CQRS read model in MongoDB for fast, flexible status timelines optimized for high‑frequency queries.

Data Stores

MySQL for transactional integrity; MongoDB for flexible, schema‑less audit and update streams.

Common questions

How is data secured?

All endpoints enforce JWT authentication, data is encrypted in transit (TLS) and at rest, RBAC controls access, and audit logs track every action for compliance.

What happens if a service fails?

Kafka retains events, consumers are idempotent, circuit breakers prevent cascades, and Kubernetes restarts failed pods automatically. No data is lost.

Can it handle millions of complaints?

Yes. Horizontal scaling via Kubernetes, Kafka partitioning, and async processing allow the system to handle massive spikes without degradation.

Does it integrate with legacy systems?

Yes. The API gateway exposes REST endpoints, and adapters can bridge SOAP, FTP, or database sync for gradual migration.

How fast is acknowledgment?

Under 100ms. The system immediately returns HTTP 202 with a ticket ID while background services handle async processing.
